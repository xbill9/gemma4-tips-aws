### 💡 AWS Inferentia2 Deployment & Cost Characteristics
AWS Inferentia2 chips are purpose-built for deep learning inference workloads and offer the best cost-per-token performance for the Gemma 4 family. 

> [!IMPORTANT]
> **Cost Discipline & Active Instance Boundary**:
> To ensure budget compliance and prevent excessive cloud spending, **never run more than one AWS EC2 host instance concurrently per project per region**. 
> Always verify active instances using `status_ec2` before provisioning new hardware, and immediately terminate redundant hosts.

* **Cost Optimization**: Inferentia2 delivers ~40% cost savings over equivalent GPU instances for steady-state, high-volume inference.
* **Trade-offs**: 
  * Model compilation tracing takes 15–30 minutes on initial startup (unless pre-compiled artifacts are loaded via `NEURON_COMPILED_ARTIFACTS`).
  * Not all quantization formats are supported yet (recommend `neuron_quant` or native INT8/FP8 configurations).
* **Instance Sizing Guidelines**:
  * **`inf2.xlarge` / `inf2.8xlarge`**: Best for smaller parameters (e.g., `google/gemma-4-12B-it` with quantization).
  * **`inf2.24xlarge` / `inf2.48xlarge`**: Required for larger models like `google/gemma-4-31B-it` to meet the high memory footprints of broader parameter counts and larger KV caches.

## 📖 vLLM User Guide for NxD Inference

[vLLM](https://docs.vllm.ai/en/latest/) is a popular library for LLM inference and serving utilizing advanced inference features such as continuous batching. This guide describes how to utilize AWS Inferentia and AWS Trainium AI accelerators in vLLM by using NxD Inference (`neuronx-distributed-inference`).

### 🔍 Overview
NxD Inference integrates with vLLM by using **vLLM's Plugin System** to extend the model execution components responsible for loading and invoking models within vLLM's LLMEngine. This means input processing, scheduling, and output processing follow the default vLLM behavior.

### 🚀 Usage Quickstart

Here is a basic python example to get started:
```python
from vllm import LLM, SamplingParams

if __name__ == '__main__':
    # Initialize the model
    llm = LLM(
        model="TinyLlama/TinyLlama-1.1B-Chat-v1.0",
        max_num_seqs=4,

        "The president of the United States is",
```

### 🗓️ Scheduling, K/V Cache, & Decoding

#### Scheduling & K/V Cache
NxD Inference uses a contiguous memory layout for the K/V cache instead of PagedAttention support. It integrates into vLLM's block manager by setting the block size to the maximum length supported by the model and allocating one block per maximum number of sequences configured. 

AWS Neuron disables preemption checks done by the scheduler in our fork. Since NxD Inference's implementation ensures each block is big enough to fit the maximum model length, preemption is never needed, which significantly improves E2E performance.

#### On-Device Sampling
On-device sampling is enabled by default to perform sampling logic directly on Neuron devices instead of passing logits to the CPU. 
- Currently supports `temperature`, `top_k`, and `top_p`.
- When `top_k` is set to -1, it is capped to 256.
- When `temperature` is 0, greedy decoding is used.

#### Quantization
NxD Inference supports quantization natively. **Do not** set vLLM's `--quantization` setting to `neuron_quant` at command-line level; keep it unset and use the Neuron model configuration to configure quantization directly.

#### Loading Pre-compiled Models & Serialization
Tracing and compiling a model can take a non-trivial amount of time (e.g., ~15 mins for 15GB). 

This is controlled via the `NEURON_COMPILED_ARTIFACTS` environment variable:
- When pointing to a path with a pre-compiled model, vLLM loads the compiled artifacts directly.
- If loading fails or is not set, a subdirectory `neuron-compiled-artifacts` will be created inside the model checkpoint folder to cache compiled weights.

### ⚠️ Known Issues & Pitfalls

1. **Chunked Prefill**: Chunked prefill is not supported on Neuron.
2. **num_gpu_blocks_override**: Must be provided to avoid out-of-bounds (OOB) errors.
   - *With blockwise KV layout (chunked prefill / prefix caching)*: Set `--num-gpu-blocks-override` to at least `ceil(max_model_len / block_size) * max_num_seqs`.
   - *With contiguous KV layout*: Set `--num-gpu-blocks-override` to exactly `max_num_seqs`.
3. **Word Embedding Tying (Qwen Models)**: When using HuggingFace IDs with shard-on-load enabled, models with `tie_word_embeddings=true` will fail with `NotImplementedError: Cannot copy out of meta tensor`. Workaround: Download the checkpoint locally and load from a local path.
4. **Async Tokenization**: Async tokenization in vLLM V1 may result in increased TTFT for small inputs or low batch sizes due to orchestration overhead.
5. **Legacy v0 Features**: Features like disaggregated inference, mllama, and speculative decoding with draft models are only supported on the legacy Neuron fork.

### 🔍 Troubleshooting & Common Issues
- **Endpoint Exits Immediately**: Verify your `--tensor-parallel-size` matches the physical Neuron cores available (e.g., `2` for `inf2.8xlarge`, `12` for `inf2.48xlarge`).

## 🧩 Alternating Local-Global Attention (Hybrid Attention) Serving Guidelines

To run models with alternating sliding-window (local) and global attention layers (like Gemma or DeepSeek-based architectures) on AWS Inferentia or similar hardware, you need to bypass standard FlashAttention hard limits and accommodate heterogeneous head dimensions (\(head\_dim = 256\) alternating with \(head\_dim = 512\)).

Follow these steps to configure and fix your vLLM pipeline:

### 1. The Core Issue
Standard attention kernels (such as FlashAttention-2) are typically hard-capped at a \(head\_dim \leq 256\). Because Gemma's global layers utilize a \(head\_dim = 512\), vLLM's standard CUDA execution drops or errors out on these specific layers. Furthermore, vLLM's KV page-size unification breaks when attempting to unify cache tensors across layers with varying dimensions.

### 2. The Configuration Fix
To resolve the head_dim and KV-cache unification errors, you must explicitly force vLLM to use custom or triton-based backends while applying block padding, or utilize LCM-based dynamic sizing.

Use the following Python vLLM runtime arguments to override standard FlashAttention behavior and circumvent backend errors:

```python
from vllm import LLM, SamplingParams

# Force the triton backend or an attention implementation that supports mixed head_dims
llm = LLM(
    model="your-gemma-model-path",
    # Override automatic backend detection to avoid FA2 head_dim limits
    attention_backend="triton", 
    # Enable dynamic block padding for alternating head dims if on vLLM v0.6+ or higher
    enforce_eager=True, 
    # Allocate custom KV cache sizing (if using int8_per_token_head) to bypass alignment crashes
    kv_cache_dtype="auto", 
)
```

### 3. Alternative Serving Options
If you are still hitting roadblocks with the vLLM native backend on Inferentia (which inherently relies on custom_ops and might suffer from throughput degradation on Triton), the architectural workaround is to use an alternative inference runtime known for robust sliding-window (SWA) support:

* **Switch to SGLang**: SGLang is optimized to natively support alternating local-global attention. It handles the heterogeneous \(head\_dim\) shifts in the Gemma 2/4 families without artificially restricting context lengths or crashing during cache allocation.

### 4. Hardware Specifics (AWS Inferentia)
If deploying on AWS Inferentia (Inf1/Inf2), ensure that your `neuronx-cc` compiler can compile custom operations. Because some proprietary Gemma kernels may fail to JIT-compile on Inferentia's neuron stack, you may need to export the vLLM models using PyTorch/TorchDynamo and apply Neuron-supported operations.
